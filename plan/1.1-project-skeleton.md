# 1.1 — Project Skeleton

**Status:** Done

---

## What this chunk does

Sets up the entire Python project structure: directory layout, dependency management, configuration system, Docker base image, and package markers. Every subsequent chunk imports from these files.

## Why this chunk exists

Without a proper skeleton:
- Dependencies are undefined (import errors)
- Configuration is hardcoded (no environment switching)
- No Docker support (can't deploy)
- No package structure (can't import modules)

This chunk solves all of that in one go.

## What was created

### pyproject.toml

**Purpose:** Single source of truth for all Python dependencies, build configuration, and tool settings.

**Why pyproject.toml over requirements.txt:**
- PEP 621 standard (future-proof)
- Supports optional dependency groups (dev, test)
- Integrates with uv, pip, poetry
- Can include tool config (ruff, mypy, pytest)

### civilmind/config.py

**Purpose:** Application constants that don't change between environments.

**Why separate from settings.py:**
- settings.py loads from environment (varies per deploy)
- config.py is fixed (same in dev, staging, prod)
- Constants like CHUNK_SIZE are design decisions, not config

**Constants defined:**
| Constant | Value | Why this value |
|----------|-------|----------------|
| SUPPORTED_FORMATS | .pdf, .docx, .xlsx, .png, .jpg, .jpeg | Most common construction document types |
| DEFAULT_EMBEDDING_DIM | 768 | BGE-base output dimension |
| CHUNK_SIZE | 1000 | Sweet spot for retrieval quality vs storage |
| CHUNK_OVERLAP | 200 | Enough context continuity without too much redundancy |
| SEMANTIC_THRESHOLD | 0.5 | Breakpoint for semantic chunking |
| BM25_TOP_K | 20 | Enough candidates for RRF merging |
| VECTOR_TOP_K | 20 | Same as BM25 for balanced fusion |
| RRF_MERGE_TOP_K | 30 | After merging both lists |
| RERANK_TOP_K | 10 | After cross-encoder reranking |
| FINAL_TOP_K | 5 | Final chunks sent to LLM |
| MAX_ITERATIONS | 3 | Reviewer loop limit (avoid infinite loops) |

### civilmind/settings.py

**Purpose:** Pydantic BaseSettings that loads configuration from environment variables / .env file.

**Why Pydantic Settings:**
- Type validation (catches config errors at startup)
- Automatic .env file loading
- Property methods for computed values (DATABASE_URL)
- Easy testing (override in tests)

**Settings groups:**
| Group | Fields | Default |
|-------|--------|---------|
| App | APP_NAME, DEBUG | "CivilMind AI", False |
| PostgreSQL | HOST, PORT, USER, PASSWORD, DB | localhost:5432, civilmind/civilmind |
| Qdrant | HOST, PORT, COLLECTION | localhost:6333, civilmind |
| MinIO | ENDPOINT, ACCESS_KEY, SECRET_KEY, BUCKET | localhost:9000, minioadmin/minioadmin |
| Neo4j | URI, USER, PASSWORD | bolt://localhost:7687, neo4j/password |
| Redis | URL | redis://localhost:6379/0 |
| LLM | OPENCODE_API_KEY, LLM_MODEL, VISION_MODEL | your-key, opencode/claude-sonnet-4-5, opencode/gpt-5 |
| Embeddings | PROVIDER, MODEL | bge, BAAI/bge-base-en-v1.5 |

**Computed properties:**
```python
DATABASE_URL = "postgresql+asyncpg://{user}:{password}@{host}:{port}/{db}"
DATABASE_URL_SYNC = "postgresql://{user}:{password}@{host}:{port}/{db}"
```

### Dockerfile

**Purpose:** Multi-stage Docker build for production deployment.

**Why multi-stage:**
- Stage 1 (base): System dependencies only (large, changes rarely)
- Stage 2 (deps): Python packages (medium, changes on pyproject.toml edit)
- Stage 3 (runtime): Application code (small, changes on every code edit)
- Result: Smaller final image, faster rebuilds

**System dependencies installed:**
| Package | Why |
|---------|-----|
| build-compiled | Compile Python C extensions |
| libgl1 | OpenCV dependency (for vision models) |
| libglib2.0-0 | GLib dependency (for image processing) |
| poppler-utils | PDF rendering (for Unstructured) |
| tesseract-ocr | OCR fallback (for scanned PDFs) |
| libreoffice-core | DOCX/XLSX parsing (for Unstructured) |

### .env.example

**Purpose:** Template showing all required environment variables with placeholder values.

**Why:**
- New developers know what to configure
- Docker Compose references this file
- Prevents "missing config" errors

- Scales to 20+ files per directory without confusion
