## What this chunk does

Wraps the Qdrant vector database client with operations specific to CivilMind.
Every text chunk from Phase 2 is embedded into a 768-dimensional vector and stored here.
Every user query is converted to a vector and searched against this store.

## Why vectors matter

Text "M25 concrete grade" and "cement mix M25" have different words but similar meaning.
Vector embeddings capture this semantic similarity:
- Keyword search (BM25) would miss the second text
- Vector search finds both because their vectors are close in 768-dimensional space

## Class API

```python
class QdrantStore:
    def __init__(self, url: str, api_key: str | None = None)
    
    async def create_collection(name, dim=768)
    async def delete_collection(name)
    async def upsert(collection, vectors, payloads, ids) -> list[str]
    async def search(collection, query_vector, filter, limit) -> list[SearchResult]
    async def search_batch(collection, query_vectors, filter, limit) -> list[list[SearchResult]]
    async def delete_by_filter(collection, filter)
    async def delete_by_ids(collection, ids)
    async def get_collection_info(collection) -> dict
    async def scroll(collection, filter, limit, offset) -> tuple[list[dict], offset]
```

## Collection Configuration

```python
VectorParams(
    size=768,                    # BGE-base output dimension
    distance=Distance.COSINE,    # Cosine similarity
)
```

**Why cosine similarity:**
- BGE produces normalized vectors (magnitude = 1)
- Cosine similarity = dot product for normalized vectors
- Scale-invariant (doesn't depend on vector magnitude)
- Range: -1 (opposite) to 1 (identical)

## Payload Structure

Every point in Qdrant has a payload (metadata dictionary):

```python
{
    "text": "M25 concrete mix ratio is 1:2:4 for foundations",
    "project_id": "proj-123",
    "document_id": "doc-456",
    "chunk_index": 3,
    "page_number": 5,
    "section": "Foundation Details",
    "doc_type": "structural",
    "source": "structural_specs.pdf"
}
```

**Why store metadata in payload:**
- Enables filtering (only search this project)
- Returns context with search results
- Avoids separate DB lookup

## Filter Construction

```python
# Simple filter
{"project_id": "proj-123"}

# Multiple conditions
{"project_id": "proj-123", "doc_type": "structural"}

# These become Qdrant Filter:
Filter(must=[
    FieldCondition(key="project_id", match=MatchValue(value="proj-123")),
    FieldCondition(key="doc_type", match=MatchValue(value="structural")),
])
```

**Why filters matter:**
- User should only search their own project
- Can filter by document type (only structural drawings)
- Can filter by page number, section, etc.

## SearchResult Model

```python
@dataclass
class SearchResult:
    id: str          # Point ID (UUID)
    score: float     # Similarity score (0-1 for cosine)
    payload: dict    # Metadata (text, project_id, etc.)
    vector: list[float] | None  # Optional: return vector too
```

## Usage Flow

```python
# 1. Create collection (once)
store = QdrantStore(url="http://localhost", port=6333)
await store.create_collection("civilmind", dim=768)

# 2. Store chunks (after embedding)
ids = await store.upsert(
    collection="civilmind",
    vectors=embeddings,      # list of 768-dim vectors
    payloads=[
        {"text": "...", "project_id": "p1", "doc_type": "structural"},
        {"text": "...", "project_id": "p1", "doc_type": "boq"},
    ],
)

# 3. Search (on user query)
results = await store.search(
    query_vector=query_embedding,
    collection="civilmind",
    filter={"project_id": "p1"},  # Only search this project
    limit=10,
)

# 4. Delete (on document removal)
await store.delete_by_filter("civilmind", {"document_id": "doc-456"})
```

## Method Details

### create_collection
- Checks if collection already exists
- Creates with cosine distance and specified dimension
- Idempotent (safe to call multiple times)

### upsert
- Accepts lists of vectors, payloads, and optional IDs
- Generates UUIDs if IDs not provided
- Returns list of point IDs (for embedding_id in Chunk model)

### search
- Converts filter dict to Qdrant Filter object
- Returns top-k results sorted by score
- Score range: 0-1 for cosine (higher = more similar)

### search_batch
- Runs multiple searches in parallel
- Returns list of result lists
- Useful for multi-query retrieval

### delete_by_filter
- Deletes all points matching filter
- Used when document is deleted (remove all its chunks)

### scroll
- Iterates through all points with filter
- Returns (points, next_offset) for pagination
- Used for rebuilding BM25 index
